# PyTorch 语音通话情绪识别设计

## 1. 项目目标

构建一个可独立发布到 GitHub 的开源 PyTorch 教学案例。用户可以在本地网页上传普通话通话音频，或直接使用浏览器录音，并获得整体情绪、四类情绪分布和分段情绪时间轴。

项目采用预训练模型进行本地推理，不包含模型训练、账号系统、数据库、云端推理或分析历史。代码和页面以中文为主，代码提供详细中文教学注释，同时补充英文 README 便于国际开源协作。

## 2. 用户与使用边界

主要用户是希望学习 PyTorch 音频推理的中文开发者，以及需要本地演示语音情绪识别流程的技术人员。

首版边界如下：

- 主要面向普通话语音。
- 支持文件上传和浏览器录音。
- 显示中性、开心、愤怒、悲伤四类结果。
- 显示整体摘要和分段时间轴。
- 不区分说话人，不进行声纹识别。
- 不用于医疗判断、员工绩效、风控、自动处罚或其他高风险决策。
- 单文件最大 50 MB，音频最长 5 分钟。

## 3. 技术选型

项目采用 FastAPI、原生 HTML/CSS/JavaScript 和 PyTorch HuBERT。FastAPI 提供本地 API 和静态页面托管，原生前端避免额外 Node 构建链，同时保留对录音、波形、时间轴和响应式布局的完整控制。

默认模型为 `xmj2002/hubert-base-ch-speech-emotion-recognition`。模型卡声明 Apache-2.0 许可证，基于 `TencentGameMate/chinese-hubert-base`，并使用 CASIA 中文情绪语音数据训练。模型原生输出愤怒、恐惧、开心、中性、悲伤和惊讶六类。模型权重约 1.1 GB，首次运行从 Hugging Face 下载，之后使用本机缓存，不提交到本项目仓库。

项目许可证为 MIT。第三方模型、前端图标、FFmpeg 运行组件和其他依赖的许可证在第三方声明中分别列出。

## 5. 音频处理与推理

### 5.1 输入和解码

前端接受 WAV、MP3、FLAC、OGG 和 M4A，后端根据内容和实际解码结果再次校验，不单独信任扩展名或 MIME 类型。压缩格式通过随 Python 依赖提供的 FFmpeg 可执行组件解码，不要求用户自行安装系统 FFmpeg。

上传内容先执行流式大小限制。需要落盘解码时使用操作系统临时目录和随机文件名，原始文件名不进入日志；临时文件在 `finally` 中删除。解码后的波形转换为 `float32`、单声道、16 kHz。包含 NaN、Inf、零采样或超过时长限制的内容立即拒绝。

### 5.2 分段

音频使用 6 秒窗口和 5 秒步长切分，尾段保留并由特征处理器填充。每段计算 RMS；低于配置阈值的段标记为静音，不送入模型。分段函数返回稳定的起止时间、有效样本数、能量和波形，便于独立测试。

### 5.3 模型

`model.py` 实现与权重匹配的 HuBERT 分类头，使用 Hugging Face 配置和特征处理器加载权重。模型采用延迟单例加载，启动页面时不阻塞下载。设备优先级为 CUDA、MPS、CPU，模型进入 `eval()`，推理使用 `torch.inference_mode()`。

CUDA 显存不足时，服务释放 CUDA 缓存并在 CPU 上完整重试当前请求一次。模型加载失败、配置不匹配或输出维度错误均返回受控错误，不把堆栈或本机路径发送到页面。

## 6. 四分类投影和可靠性

模型原始六类概率始终保留在后端分析过程，用于判断四分类是否覆盖当前语音。前端目标概率由愤怒、开心、中性和悲伤四个原始概率重新归一化：

```text
target_probability[i] = raw_probability[i] / sum(raw_target_probabilities)
excluded_probability = probability[fear] + probability[surprise]
```

若目标概率总和接近零，则该段不产生四分类结论。以下任一条件成立时，结果标记为低可靠性：

- 排除概率大于 35%。
- 四分类最高概率低于 45%。
- 四分类前两名概率差小于 12%。

低可靠性结果仍显示概率，但主文案使用“更倾向于”，并明确当前四分类覆盖不足。该规则避免把恐惧或惊讶强行伪装为四个目标类别之一。

整体概率由有效分段按有效时长与裁剪后的语音能量共同加权。能量权重设置上下限，避免极大音量主导整段结果。整体可靠性同时考虑加权后的排除概率、类别置信度、前两名差值和有效语音覆盖率。全静音或没有有效分段时不返回情绪判断。

## 7. API 设计

### 7.1 端点

- `GET /api/health`：返回服务、模型和设备状态，不触发模型下载。
- `POST /api/model/load`：显式触发模型加载，返回加载状态。
- `POST /api/analyze`：接收 multipart 音频并返回完整分析结果。

### 7.2 分析响应

响应包含：

- 整体主导情绪、四类概率、可靠性级别和可靠性原因。
- 整体排除概率、有效语音占比、音频时长、设备和耗时。
- 每个分段的起止时间、四类概率、排除概率、可靠性和静音状态。
- 稳定的请求状态与错误码。

响应不回传原始文件名、临时路径、音频字节、模型缓存路径或内部异常信息。

## 8. 页面设计

页面名为“声析 · 通话情绪识别”。它是明亮、紧凑的中文工作台，不使用营销式首屏、渐变背景、装饰光斑、超大标题或嵌套卡片。

视觉使用暖白背景、墨黑正文和朱红主操作色。四类情绪分别使用青灰、竹绿、朱红和靛蓝，另使用琥珀色提示低可靠性。卡片圆角不超过 8 px，阴影轻且只用于确实悬浮的控件。字体采用系统中文字体栈，不依赖外部字体网络请求。

顶部紧凑标题栏显示项目名、本地服务状态、模型状态和推理设备。桌面主体为左右两栏：左侧音频工作区，右侧结果区；移动端改为单列。

音频工作区使用“上传音频 / 现场录音”分段控件。上传后展示文件信息、播放器和 Canvas 波形。录音区提供录音、暂停、继续、结束和重录，并把浏览器 PCM 数据编码为 WAV，降低服务端格式差异。底部提供一个明确的“开始分析”主按钮和图标式清除按钮。

结果区展示主导情绪、可靠性、耗时、四类概率条和分段时间轴。选择时间轴片段会定位播放器并显示该段概率；播放时同步高亮当前片段。加载、空白、成功和失败状态使用固定最小高度，避免布局跳动。

移动端按钮和时间轴满足触控尺寸，无水平页面滚动；长文件名截断并保留完整提示。所有图标使用项目本地分发的 Lucide SVG 资源，不调用 CDN。

## 9. 状态与异常

前端使用单一状态机管理空白、音频已就绪、录音中、模型加载中、分析中、成功和失败状态，禁止互相冲突的操作。进行分析时显示当前分段进度；清除操作停止播放、释放对象 URL、重置录音缓冲并清空结果。

后端提供稳定错误码，覆盖空文件、文件过大、格式不支持、解码失败、音频过长、音频无效、全静音、模型下载失败、模型加载失败、显存不足重试失败和推理失败。错误文案简短、中文化并与相关控件相邻，不使用浏览器原生弹窗。

服务日志使用结构化事件名，只记录错误码、时长区间、分段数量、设备和耗时等非识别信息。默认不记录客户端地址、User-Agent、原始文件名、音频内容或预测结果。

## 10. 隐私和安全

网页只连接本机 FastAPI 服务，推理不调用远程 API。除首次模型下载外，分析流程无需互联网。项目不保存音频、结果或历史记录，不使用账号、Cookie、遥测和分析脚本。

仓库不得包含真实通话录音、个人姓名、私人邮箱、手机号、账号标识、访问令牌、环境文件、绝对路径或模型缓存。测试只使用程序生成的正弦波、静音和随机波形。

后端限制请求大小、时长、并发分析数量和解码超时。外部进程参数使用固定列表调用，不拼接 Shell 字符串。静态服务不暴露项目目录，错误响应不包含堆栈。

## 11. 测试与验收

### 11.1 自动化测试

- `test_audio.py`：解码、单双声道、重采样、分段边界、尾段、静音、NaN 和时长限制。
- `test_model.py`：延迟加载、`eval()`、推理模式、标签顺序、设备选择、错误包装和 CPU 回退。
- `test_analyzer.py`：四分类归一化、排除概率、三个可靠性阈值、静音排除和加权聚合。
- `test_api.py`：健康检查、模型状态、合成音频成功路径和全部公开错误码。
- `test_privacy.py`：响应、日志、Git 跟踪文件和示例内容不包含文件名、路径或敏感模式。

常规测试注入轻量假模型，不访问网络、不下载权重。单独的真实模型冒烟脚本用于人工验收首次下载和单段推理。

### 11.2 页面验收

Playwright 在桌面和手机视口检查：

- 上传和录音模式切换。
- 音频信息、播放器与波形布局。
- 模型加载、分析进度、失败和成功状态。
- 四类概率条和分段时间轴交互。
- 文本溢出、按钮触达、遮挡和页面滚动。
- 模拟媒体流下的录音状态机。

### 11.3 验收标准

- 新用户可按照 README 在受支持的 Python 环境中安装并启动页面。
- 上传或录制有效普通话音频后可获得整体四分类结果和分段时间轴。
- 页面在桌面和手机视口精美、稳定、符合中文工具操作习惯。
- 代码包含有教学价值的详细中文注释。
- 常规测试与 GitHub Actions 不下载真实模型并全部通过。
- 真实模型冒烟测试能够完成一次本地 PyTorch 推理。
- 音频与结果不持久化，也不发送到远程推理服务。
- 仓库内容与 Git 历史通过隐私和敏感信息扫描。
